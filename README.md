# ИССЛЕДОВАТЕЛЬ СОЛНЕЧНОЙ СИСТЕМЫ (3D)

Интерактивное веб-приложение на Three.js/Vite для визуализации Солнечной системы: планеты, карликовые планеты, транснептуновые объекты, спутники, пояса малых тел, реальные астероиды и кометы.

## 1) Технологии

- JavaScript (ES Modules)
- Three.js (рендер 3D-сцены, материалы, геометрия, камеры, свет)
- Three.js postprocessing
- `OrbitControls` для управления камерой
- Vite для dev/build
- HTML/CSS для интерфейса и карточек информации
- HTTP API NASA/JPL для части динамических объектов

## 2) Как запустить

```bash
npm install
npm run dev
```

Сборка production:

```bash
npm run build
```

## 3) Архитектура приложения

Основная логика находится в `main.js`:

- инициализация сцены, камеры, рендера, освещения;
- создание Солнца, звёздного фона и «дальних звёзд»;
- генерация поясов астероидов/объектов внешней системы;
- создание массива `celestialBodies` (основные тела + метаданные);
- построение мешей планет/карликовых планет/ТНО;
- создание спутников как дочерних `pivot`-узлов планеты;
- анимационный цикл `animate()` (орбиты, осевое вращение, обновление UI, bloom);
- интерактивность: клики, карточка объекта, режимы слежения, горячие клавиши.

## 4) Откуда берутся данные

В проекте используются два типа данных:

1. **Локальные**
   - Основные характеристики тел: название, тип, описание, диаметр, период, расстояние в а.е., год открытия, список спутников.
   - Эти данные хранятся прямо в массиве `celestialBodies`.

2. **Сетевые (NASA/JPL, частично в реальном времени)**
   - NASA NEO API: `https://api.nasa.gov/neo/rest/v1/neo/browse`
   - JPL Small-Body DB API: `https://ssd-api.jpl.nasa.gov/sbdb.api`
   - JPL Sentry API: `https://ssd-api.jpl.nasa.gov/sentry.api`
   - JPL CAD API (кометы): `https://ssd-api.jpl.nasa.gov/cad.api?body=COM`

Важно: динамические API-данные применяются только для части объектов (кометы/астероиды и т.п.). Основная «энциклопедическая» информация о телах системы задаётся локально.

## 5) Как считаются орбиты и скорости

### 5.1 Орбиты планет и карликовых тел (внутренняя модель сцены)

Для каждого тела создаётся `pivot`:

- меш тела ставится на `mesh.position.x = body.dist`;
- орбитальное движение создаётся вращением `pivot.rotation.y += body.speed * realTimeMultiplier` в `animate()`.

То есть здесь используется **параметрическая визуальная модель**: `dist` и `speed` заданы вручную в сцене и оптимизированы под наглядность.

### 5.2 Спутники вокруг планет

Для каждого спутника создаётся отдельный `moonPivot`, который является дочерним объектом меша планеты:

- `moonMesh.position.x = moonData.dist`;
- в анимации: `moon.pivot.rotation.y += moon.speed * 0.1 * realTimeMultiplier`.

Из-за родительской иерархии спутник всегда движется вместе с планетой и одновременно по своей локальной орбите.

### 5.3 Скорость времени в симуляции

- переменная `animationSpeed` управляется слайдером;
- множитель `realTimeMultiplier` влияет на все вращения;
- при `animationSpeed = 0` сцена может работать в «реальном» режиме обновления времени.

### 5.4 Астероидные пояса и внешние популяции

Объекты создаются процедурно:

- каждому объекту назначаются `angle`, `radius`, `orbitSpeed`, `rotationSpeed`;
- в `animate()`:
  - `angle += orbitSpeed * realTimeMultiplier`;
  - `x = cos(angle) * radius`, `z = sin(angle) * radius`.

Это приближённая визуализация.

### 5.5 Реальные астероиды (по элементам орбиты)

Для части астероидов используется преобразование Кеплера через функцию `keplerToCartesian()`:

- вход: `a, e, i, om, w, ma` (из SBDB/JPL);
- решение для эксцентрической аномалии итерацией;
- переход в 3D-декартовы координаты;
- масштабирование `AU_TO_SCENE`.

Это даёт более «физичный» расчёт положения по орбитальным элементам, чем у чисто процедурных поясов.

## 6) Как устроено управление и UI

- `OrbitControls`: вращение/зум/панорама камерой.
- Режимы слежения:
  - за планетой (`followPlanet`),
  - за спутником (`followMoon`),
  - за Солнцем (`followSun`),
  - остановка слежения (`stopFollowingPlanet`).
- Карточка объекта показывает:
  - период,
  - относительный размер,
  - расстояние от Солнца,
  - год открытия,
  - описание,
  - список спутников.
- Включаемые слои: орбиты, спутники, реальные астероиды, кометы, отдельные группы поясов.
- Подписи: отдельное переключение подписей планет и спутников.

## 8) Ограничения модели

- Масштабы расстояний/размеров и скорости подогнаны под визуализацию, а не под строгий 1:1.
- Не все объекты Солнечной системы включены (показана выбранная подмножина + процедурные популяции).
- Динамические API-данные зависят от доступности внешних сервисов и сетевого соединения.

## 9) Кратко о точности

Проект сочетает:

- образовательную визуализацию;
- частично физически мотивированные расчёты (для части астероидов);
- процедурную генерацию больших популяций малых тел.

Это делает приложение удобным для изучения структуры Солнечной системы, при этом оно не является полноценным научным пакетом.
